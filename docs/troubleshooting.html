Troubleshooting build errors

How to debug problems in the docs system.

First steps

If you're having an issue not explicitly listed on this page, and it's only happening locally, follow these instructions.

  1. Run git status -- do you have any changes?
    • If you do, let's start clean: git stash (read more about git stash).
  2. Run node --version -- this version should match the version listed in .nvmrc.
    • If it doesn't, and you're using nvm, run nvm use.
    • If you're not using nvm or any node version manager, it is suggested! It makes jumping between projects that use different node versions really easy. See these instructions for getting started with nvm.
  3. Alright, so you have the correct node version and a clean git state. Let's clean out the cache by running rm -rf .cache.
  4. Now clean out your node_modules with rm -rf node_modules.
  5. Let's reinstall your node_modules now that we're on the right node version: yarn.
  6. Finally, yarn dev -- are you still getting errors? Then jump into #next-docs in slack and we'd love to help you!

Setup issues

The problems in this section occur when you try to start the docs system.

SSO errors

If you receive an error message like this when you start the docs system:

remote: The `elastic' organization has enabled or enforced SAML SSO. To access
remote: this repository, visit https://github.com/orgs/elastic/sso... and try your request again.

This problem can occur if your GitHub token isn't authorized to work with SSO.

In GitHub, go to Settings> Developer settings > Personal access tokens. Click Enable SSO beside the token that you use in the docs system.

Node modules errors

If you get a build error referencing node_modules, there's likely an issue with your local node_modules directory.

To fix the error, run the following commands one at a time, in order.

  1. cd into your local elastic-docs repo.
  2. Pull the latest changes from master branch of the elastic/elastic-docs repo.
  3. Run rm -rf .cache.
  4. Run rm -rf node_modules.
  5. Run yarn and then yarn dev.

Missing module errors

If you get a build error because of a missing module like this one, run yarn again from the root of the elastic-docs repo to pick up and install the missing module dependency:

ERROR #11321  PLUGIN

"micromark-extension-mdxjs" threw an error while running the sourceNodes lifecycle:

Cannot find module 'micromark-extension-mdxjs'

You might run into this scenario when you've pulled down the latest files but haven't tested in a while.

Content issues

The problems in this section occur when you've created or updated a page and it's not displaying properly in the docs system.

Updates not reflected in your local environment

If you're making changes to files in your local (development) sources and you don't see those changes reflected in the browser when you refresh the page, check your sources-dev.json file. You must use a "type": "file" for your local changes to be picked up and the "location" must be a path that is relative to the location of the Elastic docs system.

If you accidentally have the same source twice in your json file (e.g. both as a "file" and "github" type of source), this also can cause your changes to be unheeded.

Curly brackets cause page failure

If you receive an error like this when navigating to a documentation page:

TypeError: Cannot read property 'getElementsByTagName' of null

or

ReferenceError: n is not defined

This problem can occur when the doc page has curly brackets {any_text_here} that are not in a code snippet. In particular, this happens when you copy content that contained shared attributes in the old documentation system. Remove the curly brackets (or replace the shared attribute with the appropriate text) and refresh the page to fix the problem.

Asciidoc internal links cause page failure

If your page contains links that were created for Asciidoc (also known as cross references), they do not work in this docs system.

For example, <<test>> results in the following error when you try to load the page:

TypeError:Cannot read property 'frontmatter'of null

To fix this situation, use the appropriate syntax per MDX docs syntax.

The GitHub link doesn't seem right

If the "Edit this page on GitHub" link isn't linking you to the right place, check your sources-dev.json. If you are pointing at local docs, this is a known, and currently accepted, issue. Our "Edit this page on GitHub" only works if you're pointing at a repo. If this is blocking for you, please file a bug in our elastic-docs repo.

Bad link and 404 errors

If you try to view your content locally and you encounter "bad link" or "There's not a page yet at... Preview custom 404 page" messages, first verify that the problematic page does indeed exist in your repo.

If the page does exist but it's being pulled from a repo with a lot of folders and files, you might be able to work around the problem by narrowing the scope of your local (development) sources. For example, instead of pulling in the full kibana repo in your sources-dev.json file, use a subset like this:

{
  "type": "file",
  "location": "../kibana/dev_docs"
},
{
  "type": "file",
  "location": "../kibana/src/plugins/data"
}

This is a temporary limitation.

Debugging

Next.js makes it really easy to attach a debugger when you're working on the code of the system. To start the code with debugging enabled, run yarn debug. Note that this assumes you have already cloned the content repos at least once (NODE_ENV=development yarn clone-sources).

If you use VS Code, you can then use the "Connect Debugger" launch target (under the "Run and Debug" pane) to connect to the running dev server.
