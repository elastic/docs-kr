Content sources

How to add new docs into and remove docs from a docsmobile site

What you'll learn

This page describes how to work with content sources in a docsmobile-powered site. By the end you'll understand how to configure local and remote folders of content.

Diagram

Loading content through sources.json

Your docsmobile-powered site loads in content from a special configuration file called sources.json, located in the root directory of your site. Below is an example of a configuration with a few different sources.

{
  "sources": [
    {
      "type": "file",
      "location": "../content"
    },
    {
      "type": "github",
      "location": "elastic/kibana",
      "branch": "v8.1"
    },
    {
      "type": "github",
      "location": "elastic/docsmobile",
      "subdirs": ["doc-site"]
    }
  ]
}

The above config will pull from three sources, a local /content/ folder in the site repo, and then from two remote Github repositories. Note the values you can pass below:

keyvaluecomments
type
file or github
Defines whether the location is remote or local
location
string
A repo name or filepath relative to the site.
branch (optional)
string or Array<string>
Specifies what branch you want (if other than the default) or a list of branches that you want. Branches are interchangeable with tags here.
subdirs (optional)
Array<string>
Allow-list of directories to pull from

If pointing to a remote repository, the docsmobile system will clone that repo on startup. When working with the file type, any changes you make in that directory should be picked up and show up in your site.

Working locally with sources-dev.json

Similar to the above you can set up a sources-dev.json file in your root directory. This file will take precedence when you start your docs site in yarn dev mode. A common scenario for local development is that you point at file type sibling directories that you have locally cloned instead of remote Github repos. When running yarn build or running in production mode, the sources.json file will always be used.

Custom TSX pages

Beyond using the MDX pages, you can optionally create your own custom pages using typical React TSX files. Simply create a dir in your primary site repo called /pages/ and any pages you add with replicate the file-path as a page slug. This uses the Next.js page concept under the hood.

Example: replacing the home page

To create a custom homepage for your docsmobile-powered site would create a /pages/index.tsx file. These files work like any other TypeScript template file and can include state, interaction and imports of any components you need.

In order to use components in our system (for now) to build up your page, your import will look something like this:

Due to the way the build works, imports of Docsmobile components are a little irregular and need to be ignored by TypeScript. This will be fixed soon.
import { EuiText } from "@elastic/eui";
// @ts-ignore
import { BaseLayout } from "@elastic/docsmobile/components/layouts/base_layout";
// @ts-ignore
import { DocLink } from "@elastic/docsmobile/components/doc_link";

export const Index: FunctionComponent = () => {
  return (
    <BaseLayout>
      <EuiText>
        <p>My custom homepage content will now be surrounded by the usual header and footer.</p>
        <p>I can still mix in Docsmobile components like a <DocLink id="somePageId" /></p>
      </EuiText>
    </BaseLayout>
  );
}

export default Index;

Troubleshooting

  • Changes to sources and /pages/ only happen when you restart the server.
  • Because of a bug in docsmobile (as of March 08, 2021) the location for file types needs to be one level higher than you'd expect. This usually means appending an extra ../ to the path then you'd expect.
