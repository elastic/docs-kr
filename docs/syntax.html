MDX docs syntax

Code syntax and examples for the custom widgets we use to write the docs

The Elastic documentation engine allows for markdown-like content using MDX. We often replace and augment the core markdown syntax with React-powered renders that provide additional functionality. In all cases these components are preceeded with the Docs prefix.

Frontmatter

Every MDX document needs to include the following frontmatter.

  • id (Required) Must be unique, should never change. Used in links between pages and in navigation systems. Use camelCase format, for example thisMakesSense.
  • slug (Required) Defines the URL path for the page. Must be prefixed by a slash. Should use a prefix to prevent conflicts. Use lowercase and hyphens, for example /docs/this-makes-sense.
  • title (Required) Default title display for the document in page headers and links.
  • image (Optional) Image used for cards and other graphical lists when linking to this page.
  • description (Optional) One-liner that describes the purpose of the page for human and search engines.
  • date (Optional) Last updated or verified date for the page.
  • tags (Optional) Categories that help define related pages.
  • related (Optional) An array of page IDs that complement the information in the current page.
  • layout (Optional) Override the default article layout. Passing the value landing will remove the article's table of contents from the right side of the page.

# Frontmatter starts at the top of the document

Frontmatter needs to appear on every MDX document at the start of the page.

For more details about recommended values, refer to Content Guidelines.

Links

Links come in two formats. For internal links within the site, utilize the DocLink component which will fetch the slug and title from the page's front matter. You may optionally include the tooltip prop which will provide wikipedia-style link previews.

Internal links

<DocLink id="docsSyntax" text="Different link text"/>
<DocLink id="docsSyntax" tooltip />
<DocLink id="docsSyntax" section="unordered-lists" text="Linking to a specific section" />

If you include a section qualifier, you must use the resolved anchor ID (e.g. "unordered-lists"); do not use the section title (e.g. "unordered lists").

External links

External links use regular Markdown formatting. rel="no-follow" will be automatically applied.

Variable interpolation

Variable interpolation replaces a variable with a provided word or phrase that is defined in a mapping in 'variables.json'. The key in variables.json can contain alpha-numeric characters, underscores, and hyphens, and is the variable you'll be using in the markdown. The value is what will be injected in its place. If you need to reuse something longer than a word or phrase, use partial mdx files instead.

Variable interpolation can be used by itself, within other text, or inside of a code block. For example:

Wow, I sure love ((eui))!

((isengard))

`variables even work in code blocks: ((catName))`

They're taking the hobbits to Isengard!

Lists

You can create lists using typical Markdown format.

To add another element in a list while preserving the continuity of the list, indent the element four spaces or one tab.

Definition lists

To create a definition list (also known as a description list), type the term on the first line. On the next line, type a colon followed by a space and the definition. For example:

Term 1

: Definition 1

Ordered lists

To create an ordered list, use numbers followed by periods. The numbers don’t have to be in numerical order, but the list should start with the number one. For example:

1. First item
3. Second item
5. Third item
    1. Indented item
    2. Indented item
8. Fourth item
  1. First item
  2. Second item
  3. Third item
    1. Indented item
    2. Indented item
  4. Fourth item

Unordered lists

To create an unordered list, use dashes (-), asterisks (*), or plus signs (+). Indent one or more items to create a nested list. For example:

- First item
- Second item
- Third item
    - Indented item
    - Indented item
- Fourth item
  • First item
  • Second item
  • Third item
    • Indented item
    • Indented item
  • Fourth item

Tabs

To structure your content into a tabbed interface use the DocTabs component. Individual tabs are easily created using DocTab. A tab name must be specified for each. (e.g. DocTab name='macOS').

Inside the DocTab block, you can use Markdown syntax, HTML, and other doc components.

You can add a border around the tabs to separate them from other content by adding the hasBorder prop to DocTabs.

Content for your first tab. You can add Markdown syntax and other doc components, like this link.


<DocTabs hasBorder>
  <DocTab name="First tab">
    Content for your first tab. You can add **Markdown** syntax *and* other doc components, like <DocLink id="docsSyntax" text="this link" />.
  </DocTab>
  <DocTab name="Second tab">
    Content for your second tab. Why not add a code block?

    ```html
     <p>Here is an example of some code</p>
    ```
  </DocTab>
</DocTabs>

Alternatively, you can pass the individual tabs in as an array of objects. However, it's important to note that when using this approach, Markdown cannot be used inside the content prop.

Horizontal rule

A thematic break (horizontal rule) can be added with three or more matching -, _, or * characters. A blank line must come immediately before and after the matching characters.

This paragraph is about cars.

---

This paragraph is about cats.

Landing pages

Landing pages should be used to give users a quick introduction to the content in the category or section that they're about to read. They are comprised of a prominent title and summary area at the top of the page, followed by either a paragraph of introductory text or a set of links to related articles.

The above page is very easy to create in markdown.

---
id: docsLanding
title: Elastic Docs System
description: Documentation about the docs system.
slug: /docs/landing/
layout: landing
---
import items from './landing-example.json'

<DocLandingHero
  icon='pencil'
  title='Docs System'
  description="Find out about the system behind the system. This set of documentation will help you run the new docs system locally and begin contributing to it's development."
  buttonText='Get started'
  landingPageId='docsSyntax'
/>

<DocRelatedArticles
  sectionTitle='Related articles'
  items={items}
/>

The frontmatter is consistent with other docs article pages with one exception, by adding layout: landing, the table of contents typically present on the right side of most docs pages is no longer visible.

Hero section

A hero section can focus a page around a clear objective. Try to ensure that you have provided a precise title and an informative description. The DocLandingHero components require the following attributes: title, description, and icon (as one of EUI's icon types). If you wish to add a prominent call-to-action button, you can do so by including the ctaButtonText and ctaPageId attributes.

<DocLandingHero
  icon='pencil'
  title='Docs System'
  description="Find out about the system behind the system. This set of documentation will help you run the new docs system locally and begin contributing to it's development."
  ctaButtonText='Get started'
  ctaPageId='docsSyntax'
/>

Related articles

Related articles are a great way to highlight the most important, or most popular articles in your documentation, ensuring they're discoverable to users. The DocRelatedArticles component takes an optional sectionTitle string attribute, as well as an array of items.

If you wish to link to a page outside of the documentation, rather than using pageId in the items object, href is an available prop and target can be used to specify whether the link opens in a new tab or not.

Limitations

Reserved Syntax

The characters <> and {} are reserved for components and expressions, respectively. As text, they can only be added to documentation files when in code blocks or snippets. The $ character is reserved for math equations. As text, they either need to be escaped (\$) or used in code blocks or snippets. Double (()) is reserved for variable interpolation. As text, they either need to be escaped: \(\(\)\). In a code block you only need to escape the first parenthesis.

Excluded files

MDX files meant for Storybook will not be included (i.e. files that end in .stories.mdx). An mdx page without a slug won't be created into a page (how would we know where to put it?).
