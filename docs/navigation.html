Navigation trees

Learn how to set up and manage navigation trees in docsmobile-powered sites

What you'll learn

This page describes how to work with navigation trees in a docsmobile-powered site. By the end you'll understand how to set up nested navigation for each of your content sources.

Diagram

Document tree structure

Docsmobile configures the navigation structure separately from the folder structure of the source files. Essentially, any files you imbibe--be it locally or from a GitHub repo--will come in as flat documents. It's up to you to provide a taxonomy for those files. This means that each time you add a document to the system, you must consider whether that document belongs in the navigation, and if so, add it to the shared navigation configuration file.

Not all documents necessarily have to appear in the navigation, especially if you're aiming for a limited number of levels of depth in your navigation tree. Documents that are not in the navigation can nonetheless be linked to and are returned in search results.

General concepts of navigation trees

All navigation links and structure are provided in the /nav.json file. There are a few key concepts to be aware of:

  • The top tree nav.json structure lives the root of the website. It defines your top level tree.
  • That nav.json file can reference any number of individual tree files which can live anywhere in your content directories. These are usually naming something.docnav.json.
  • Those individual tree files reference individual pages (that can live anywhere within the sources purview and are referenced by the id on those pages).

An example nav with two content sources

Here's an example navigation setup. Within the main site repo we'd include a nav.json file in the root.

// nav.json
{
  "default": "design",
  "structure": [ "design", "ml" ]
}

The structure above references two navigation trees, which themselves might live within two separate content repos defined in the sources.json file. Note, that it's not important where these files live in the folder structure, only that they are named properly with correct file names and ids.

Nav files referenced from other repos

This would be the design nav as referenced above. It might live anywhere in elastic/design-manual and would be called design.docnav.json.

{
  // Categories at this level populate the dropdown
  "mission": "Product design",
  "id": "design", // This ID is required and must be unique.
  "landingPageId": "designCharter", // This is the required id for the page where the mission should go to if clicked on its own. Could be any page id within the navigation.
  "icon": "faceHappy", // EUI icon
  "description": "Manual for how the Product Design organization is run.",
  "items": [
    // Each item object represents a sub-menu with this navigation
    {
      "label": "Organization",
      "items": [
        // These are the items within that fold. They can accept labels.
        { "label": "Custom label for charter", "id": "designCharter" },
        { "id": "designHowWeInterview" }
      ]
    },
    {
      "label": "Hiring",
      "items": [
        { "id": "designHowWeInterview" },
        { "id": "designScreening" },
        { "id": "designSkillsInterview" }
      ]
    }
  ]
}

And then we might have a separate nav located in elastic/ml-team called ml.docnav.json.

{
  "mission": "ML team docs",
  "id": "ml",
  "landingPageId": "mlTeamMembers",
  "icon": "pencil",
  "description": "Documentation for the ML team",
  "items": [
    {
      "label": "About our team",
      "items": [{ "id": "mlMembers" }, { "id": "mlCharter" }]
    }
  ]
}

Troubleshooting

  • Currently any edits made to navigation files require your dev server to restart to take effect.
